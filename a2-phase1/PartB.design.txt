Monitor Solution Strategy (Detailed)
------------------------------------------------------------

Our monitor uses UBC pthreads and the Send/Receive/Reply IPC model.
All monitor operations are centralized through a dedicated monitor server.
This server is a UBC pthreads process, created once in MonInit().
It runs an infinite loop, waiting for messages from client threads.
Each message represents a monitor operation (MonEnter, MonLeave,
MonWait, MonSignal, MonInit).
The server processes each message, prints a debug statement, and replies.
This ensures only one thread interacts with the monitor at a time.

**Key Components:**

1. **Monitor Server Process:**
   - Created using Create() in MonInit().
   - Waits for messages using Receive().
   - Handles each monitor operation in a switch statement.
   - Replies to each sender using Reply().
   - Can be extended to manage monitor state and queues.

2. **Monitor API Functions:**
   - Each function (MonEnter, MonLeave, etc.) sends a message to the server.
   - The calling thread blocks until the server replies.
   - This guarantees serialized access to the monitor.

3. **Message Structure:**
   - Each message has a type (operation) and an argument (e.g., condition).
   - The server can be extended to track more state as needed.

4. **Condition Variables:**
   - For Phase 1, MonWait and MonSignal only print debug messages.
   - In future phases, the server will manage queues for each condition.

**How the Monitor Works:**

- A thread calls MonEnter(), which sends a MON_ENTER message to the server.
- The server receives the message, prints a debug statement, and replies.
- The thread is unblocked and can proceed.
- The same pattern is used for MonLeave, MonWait, and MonSignal.
- MonInit() starts the server and initializes the monitor.

Why This Solution is Correct (Detailed)
------------------------------------------------------------

- **Mutual Exclusion:** All monitor operations go through the server.
  Only one thread can interact with the monitor at a time.
  Send/Receive/Reply ensures threads block until the server is ready.

- **Correctness of IPC:** UBC pthreads' primitives guarantee reliable
  message delivery and synchronization. The sender blocks until the
  server replies, so no two threads can execute monitor code together.

- **Extensibility:** The design is modular and can be extended to support
  full monitor semantics, such as condition variables and signaling.
  The server can be enhanced to manage queues and implement correct
  semantics for MonWait and MonSignal.

- **Debuggability and Testability:** For Phase 1, the server prints all
  received messages, making it easy to verify that monitor operations
  are serialized and IPC is working.

- **Separation of Concerns:** The monitor server handles all
  synchronization and state. Client threads only send requests and wait
  for replies. This separation makes the design easy to reason about
  and extend.
