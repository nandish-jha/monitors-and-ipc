Part A: xv6 Kernel Mutex Implementation
------------------------------------------------------------

Overview:
This document describes the design and implementation of mutex synchronization
primitives in the xv6 operating system kernel. The implementation provides
four system calls for mutex creation, locking, unlocking, and destruction,
enabling user programs to achieve mutual exclusion and synchronization.

------------------------------------------------------------
System Call Interface
------------------------------------------------------------

1. mtx_create(int flags)
   Purpose: Creates a new mutex in the kernel
   Parameters: flags (currently unused, reserved for future extensions)
   Returns: mutex ID (non-negative integer) on success, -1 on failure
   
2. mtx_lock(int mutex_id)
   Purpose: Acquires a mutex lock (blocks if unavailable)
   Parameters: mutex_id - ID of the mutex to lock
   Returns: 0 on success, -1 on failure
   
3. mtx_unlock(int mutex_id)
   Purpose: Releases a mutex lock
   Parameters: mutex_id - ID of the mutex to unlock
   Returns: 0 on success, -1 on failure
   
4. mtx_destroy(int mutex_id)
   Purpose: Destroys a mutex and frees kernel resources
   Parameters: mutex_id - ID of the mutex to destroy
   Returns: 0 on success, -1 on failure

------------------------------------------------------------
Kernel Data Structures
------------------------------------------------------------

Mutex Structure:
```c
struct mutex {
    int allocated;     // Is this mutex slot in use? (0=free, 1=allocated)
    int locked;        // Is the mutex currently locked? (0=unlocked, 1=locked)
    int owner_pid;     // PID of process that owns the lock (-1 if unlocked)
    struct proc *waiting; // Linked list of processes waiting for this mutex
};
```

Global Mutex Table:
```c
struct mutex mutexes[MAX_MUTEXES];  // Array of mutexes (MAX_MUTEXES = 64)
struct spinlock mutex_table_lock;   // Protects the mutex table from races
```

Constants:
- MAX_MUTEXES: 64 (maximum number of mutexes in the system)
- MUTEX_UNLOCKED: -1 (value for owner_pid when mutex is unlocked)

------------------------------------------------------------
Implementation Strategy
------------------------------------------------------------

1. **Mutex Table Management:**
   - Global array of mutex structures indexed by mutex ID
   - Spinlock protection for atomic table operations
   - Linear search for free slots during creation
   - Reference counting via allocated field

2. **Blocking and Wakeup Mechanism:**
   - Uses xv6's existing sleep/wakeup infrastructure
   - Processes sleep on the mutex structure address
   - Wakeup all processes waiting on mutex during unlock
   - First-come-first-served scheduling for waiting processes

3. **Ownership Tracking:**
   - owner_pid field tracks which process owns each mutex
   - Only the owning process can unlock a mutex
   - Prevents unauthorized unlocking and double-unlocking

4. **Error Handling:**
   - Validates mutex IDs in all system calls
   - Checks process ownership before unlocking
   - Returns appropriate error codes for invalid operations

------------------------------------------------------------
System Call Implementation Details
------------------------------------------------------------

mtx_create():
1. Acquire mutex_table_lock
2. Search for free slot in mutexes[] array
3. If found, initialize mutex structure:
   - allocated = 1
   - locked = 0
   - owner_pid = MUTEX_UNLOCKED
   - waiting = NULL
4. Release mutex_table_lock
5. Return mutex ID (array index) or -1 if table full

mtx_lock():
1. Validate mutex_id parameter
2. Acquire mutex_table_lock
3. Check if mutex is allocated
4. If mutex is unlocked:
   - Set locked = 1
   - Set owner_pid = current process PID
   - Release mutex_table_lock and return success
5. If mutex is locked:
   - Add current process to waiting list
   - Release mutex_table_lock
   - Call sleep() on mutex address
   - When awakened, retry the lock attempt
6. Return 0 on success, -1 on error

mtx_unlock():
1. Validate mutex_id parameter
2. Acquire mutex_table_lock
3. Check if mutex is allocated and locked
4. Verify current process is the owner
5. Set locked = 0 and owner_pid = MUTEX_UNLOCKED
6. Call wakeup() on mutex address to wake waiting processes
7. Release mutex_table_lock
8. Return 0 on success, -1 on error

mtx_destroy():
1. Validate mutex_id parameter
2. Acquire mutex_table_lock
3. Check if mutex is allocated
4. If mutex is locked, return error (cannot destroy locked mutex)
5. Set allocated = 0 to mark slot as free
6. Release mutex_table_lock
7. Return 0 on success, -1 on error

------------------------------------------------------------
Concurrency and Race Prevention
------------------------------------------------------------

1. **Spinlock Protection:**
   - mutex_table_lock protects all mutex table operations
   - Prevents races during creation, destruction, and state changes
   - Short critical sections to minimize lock contention

2. **Sleep/Wakeup Synchronization:**
   - Leverages xv6's proven sleep/wakeup mechanism
   - Avoids busy-waiting and CPU waste
   - Handles lost wakeup race conditions correctly

3. **Atomic State Transitions:**
   - All mutex state changes occur within spinlock protection
   - Prevents inconsistent intermediate states
   - Ensures ownership transfers are atomic

------------------------------------------------------------
Design Rationale and Trade-offs
------------------------------------------------------------

Advantages:
1. **Simplicity:** Leverages existing xv6 synchronization primitives
2. **Efficiency:** Minimal overhead for uncontended locks
3. **Correctness:** Uses proven sleep/wakeup for blocking
4. **Resource Management:** Bounded mutex table prevents resource exhaustion
5. **Security:** Ownership tracking prevents unauthorized operations

Trade-offs:
1. **Scalability:** Fixed-size mutex table limits concurrent mutexes
2. **Fairness:** No guarantee of fair scheduling among waiting processes
3. **Priority Inversion:** No priority inheritance mechanism
4. **Deadlock Detection:** No automatic deadlock prevention or detection

Future Enhancements:
1. Priority inheritance to address priority inversion
2. Timeout-based locking operations
3. Reader-writer locks for improved concurrency
4. Deadlock detection and recovery mechanisms
5. Per-process mutex limits for resource management
