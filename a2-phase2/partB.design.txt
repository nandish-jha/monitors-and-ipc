Monitor Implementation with Hoare Semantics
------------------------------------------------------------

This document describes the complete implementation of monitor synchronization
primitives using UBC Pthreads Send/Receive/Reply IPC mechanisms. The 
implementation provides full monitor semantics with mutual exclusion,
condition variables, and proper Hoare signaling semantics.

------------------------------------------------------------
Architecture Overview
------------------------------------------------------------

**Core Design Philosophy:**
- Centralized monitor server process handles all synchronization
- Client threads communicate via Send/Receive/Reply IPC
- Hoare monitor semantics with immediate control transfer on signal
- Urgent queue provides proper priority ordering
- Complete condition variable support with blocking and signaling

**Key Components:**

1. **MonServer Process:**
   - Dedicated UBC pthreads process managing all monitor operations
   - Infinite message processing loop with switch-based dispatch
   - Maintains monitor state: ownership, queues, condition variables
   - Implements proper Hoare semantics for signaling operations

2. **Monitor State Management:**
   - monitor_locked: Boolean indicating monitor availability
   - current_owner: PID of process currently owning monitor
   - entry_queue: Processes waiting to enter monitor
   - urgent_queue: Processes that called MonSignal() (Hoare semantics)
   - condition_queues[]: Per-condition waiting queues
   - num_conditions: Number of initialized condition variables

3. **Client API Functions:**
   - MonInit(): Initialize monitor and create server process
   - MonEnter(): Enter monitor (acquire mutual exclusion)
   - MonLeave(): Exit monitor (release mutual exclusion)
   - MonWait(): Wait on condition variable
   - MonSignal(): Signal condition variable (Hoare semantics)

4. **Message Protocol:**
   - MonMsg structure: {type, arg}
   - Message types: MON_ENTER(1), MON_LEAVE(2), MON_WAIT(3), MON_SIGNAL(4),
        MON_INIT(5)
   - Synchronous communication: Send blocks until Reply received

------------------------------------------------------------
Hoare Monitor Semantics Implementation
------------------------------------------------------------

**Traditional Monitor Properties:**
1. **Mutual Exclusion:** Only one process can be active in monitor
2. **Condition Synchronization:** MonWait blocks, MonSignal wakes
3. **Immediate Transfer:** Signaler yields control to signaled process
4. **Priority Ordering:** Urgent queue has priority over entry queue

**Queue Priority (Hoare Semantics):**
1. **Urgent Queue:** Processes that called MonSignal() await re-entry
2. **Entry Queue:** Processes waiting for initial monitor entry
3. **Condition Queues:** Processes waiting on specific conditions

**Signal Operation Semantics:**
When MonSignal(cond) is called:
1. If no process waiting on condition: signaler continues normally
2. If process waiting on condition:
   - Signaled process gets immediate monitor control
   - Signaling process goes to urgent queue (blocked)
   - Monitor ownership transfers immediately

**Monitor Entry/Exit Priority:**
When monitor becomes available (MonLeave or MonWait):
1. Check urgent queue first - highest priority
2. If urgent queue empty, check entry queue
3. Grant monitor access to first process in highest priority queue

Detailed Implementation
------------------------------------------------------------

**MonServer Message Processing:**

```c
PROCESS MonServer(void *arg) {
    while (1) {
        MonMsg *msg = (MonMsg *)Receive(&sender_pid, &msglen);
        
        switch (msg->type) {
            case MON_ENTER:
                if (monitor_locked == 0) {
                    // Grant immediate access
                    monitor_locked = 1;
                    current_owner = sender_pid;
                    Reply(sender_pid, &success);
                } else {
                    // Add to entry queue, don't reply (blocks sender)
                    ListAppend(entry_queue, &sender_pid);
                }
                break;
                
            case MON_LEAVE:
                if (current_owner == sender_pid) {
                    monitor_locked = 0;
                    current_owner = -1;
                    Reply(sender_pid, &success);
                    
                    // Hoare semantics: urgent queue priority
                    if (ListCount(urgent_queue) > 0) {
                        next_pid = dequeue(urgent_queue);
                        grant_monitor(next_pid);
                    } else if (ListCount(entry_queue) > 0) {
                        next_pid = dequeue(entry_queue);
                        grant_monitor(next_pid);
                    }
                }
                break;
                
            case MON_WAIT:
                if (current_owner == sender_pid) {
                    // Add to condition queue
                    ListAppend(condition_queues[msg->arg], &sender_pid);
                    
                    // Release monitor
                    monitor_locked = 0;
                    current_owner = -1;
                    
                    // Priority: urgent queue first, then entry queue
                    transfer_control_to_next_process();
                    
                    // Don't reply - sender blocks on condition
                }
                break;
                
            case MON_SIGNAL:
                if (current_owner == sender_pid && 
                    ListCount(condition_queues[msg->arg]) > 0) {
                    
                    // Hoare semantics: immediate transfer
                    wakeup_pid = dequeue(condition_queues[msg->arg]);
                    
                    // Signaler goes to urgent queue
                    ListAppend(urgent_queue, &sender_pid);
                    
                    // Transfer monitor to signaled process
                    current_owner = wakeup_pid;
                    Reply(wakeup_pid, &success);
                    
                    // Don't reply to signaler - they're in urgent queue
                } else {
                    // No waiting process or invalid signal
                    Reply(sender_pid, &success);
                }
                break;
        }
    }
}
```

Reader-Writer Problem Implementation
------------------------------------------------------------

**Test Case: Reader-Writer Monitor**
Using the monitor to solve the classic readers-writers problem:

```c
// Shared variables (protected by monitor)
static int numReaders = 0;
static int busyWriting = 0;

// Condition variables
#define OKtoRead  0
#define OKtoWrite 1

void StartRead() {
    MonEnter();
    while (busyWriting)
        MonWait(OKtoRead);
    numReaders++;
    MonSignal(OKtoRead);  // Wake other readers
    MonLeave();
}

void StopRead() {
    MonEnter();
    numReaders--;
    if (numReaders == 0)
        MonSignal(OKtoWrite);  // Wake waiting writer
    MonLeave();
}

void StartWrite() {
    MonEnter();
    while (numReaders > 0 || busyWriting)
        MonWait(OKtoWrite);
    busyWriting = 1;
    MonLeave();
}

void StopWrite() {
    MonEnter();
    busyWriting = 0;
    MonSignal(OKtoWrite);  // Wake waiting writer
    MonSignal(OKtoRead);   // Wake waiting readers
    MonLeave();
}
```

**Correctness Properties Verified:**
1. Multiple readers can read simultaneously
2. Writers get exclusive access (no readers or other writers)
3. No reader-writer conflicts or race conditions
4. Proper condition variable signaling
5. Clean process coordination and termination

Design Rationale and Correctness
------------------------------------------------------------

**Why This Design is Correct:**

1. **Mutual Exclusion:** 
   - Single MonServer ensures serialized monitor operations
   - Send/Receive/Reply provides atomic message processing
   - Only current_owner can perform monitor operations

2. **Hoare Semantics Correctness:**
   - MonSignal() immediately transfers control to signaled process
   - Signaling process blocks in urgent queue until re-entry
   - Urgent queue priority ensures proper signaling semantics

3. **Condition Variable Correctness:**
   - MonWait() atomically releases monitor and blocks on condition
   - MonSignal() wakes exactly one waiting process
   - Proper condition checking prevents lost wakeup problems

4. **IPC Reliability:**
   - UBC pthreads guarantees reliable message delivery
   - Synchronous communication prevents race conditions
   - Error checking ensures robust operation

**Performance Characteristics:**
- Uncontended monitor operations: O(1) message passing overhead
- Queue operations: O(1) append, O(1) remove from front
- Memory usage: Linear in number of waiting processes
- No busy-waiting: all blocking uses efficient sleep/wakeup

**Scalability Considerations:**
- Fixed-size condition variable array (MAX_CONDS = 8)
- Dynamic queue sizing based on actual usage
- Single monitor server may become bottleneck under high contention
- Design easily extensible to multiple monitor instances
